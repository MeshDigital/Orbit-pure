# Cue Forge — Architecture Reference

## Purpose

Cue Forge is ORBIT's dedicated DJ cue editor. It gives a full-track waveform view with interactive cue placement, loop editing, and structural phrase visualization. It is separate from the Workstation — the Workstation plays and mixes, Cue Forge prepares and edits the underlying cue data that gets exported to Rekordbox or loaded onto CDJs.

## Data Flow

### Track Load

```
Right-click track → TrackOperationsViewModel.ExecuteOpenInCueForge
  │
  ├─ SetPlaylistTracks(CurrentProjectTracks)    ← populates left browser sidebar
  ├─ CueForgeViewModel.LoadTrackAsync(hash, title, artist)
  │     ├─ ICuePointService.GetByTrackIdAsync   ← loads committed cues from DB
  │     ├─ ILibraryService.GetAudioFeaturesByHashAsync
  │     │     ├─ WaveformBlob [Low|Mid|High bytes] → WaveformLow/Mid/High
  │     │     ├─ EnergyCurveJson                → EnergyCurveData float[]
  │     │     ├─ VocalDensityCurveJson          → VocalDensityCurveData float[]
  │     │     ├─ BeatGridJson + OnsetDensityEngine → OnsetDensityCurveData float[]
  │     │     ├─ PhraseSegmentsJson             → PhraseSegments collection
  │     │     └─ CamelotKey                     → CurrentCamelotKey display
  │     └─ AppConfig.CueForgeLastTrackHash saved (session persistence)
  │
  └─ NavigateToPageEvent("CueForge")            ← page becomes visible
```

### Auto-Generate Cues

```
⚡ Auto-Generate button
  │
  └─ CueForgeViewModel.AutoGenerateCuesAsync
        ├─ ILibraryService.GetAudioFeaturesByHashAsync
        ├─ BuildAnalysisResultFromFeatures(features) → AnalysisPipelineResult
        │     (fast path: reconstructs signal bundle from DB cache, no audio re-decode)
        └─ Engine.Cueing.CueGenerationService.GenerateCues(...)
              │
              ├─ Path 1 (ML):  PhraseSegments.Count >= 2
              │     EDMFormer phrase segments → 8 structurally-driven cues
              │
              ├─ Path 2 (DSP): SubBassReturnTimestamps or NoveltyDropSignatures
              │     Sub-bass silence = breakdown, return = drop
              │     NoveltyDropSignature.BuildStartSeconds = build
              │     EnergyCurve tail falloff = outro
              │
              └─ Path 3 (Heuristic): transient clustering + IntentClassifier
                    Fallback when no sub-bass signals are available
```

### Cue Snapping

Every cue placement (auto or manual) goes through `SnapToBar`:
```
raw_time → SnapToBar(t, bpm, downbeatAnchor)
         = T₀ + ⌊(t - T₀) / (bar_duration) + 0.5⌋ × bar_duration
```
Where `bar_duration = 60/BPM × 4`. In the waveform control, user drags snap via `ApplySnapping` with configurable quantize resolution (Off → 128 beats).

### Commit Path

```
✓ Commit button
  │
  └─ CueForgeViewModel.CommitChangesAsync
        └─ ICuePointService.CreateManyAsync(WorkingCues → CuePointEntity[])
              Deletes all IsAutoGenerated cues for track first, then inserts new set
```

### Export to Rekordbox

```
Playlist export → RekordboxXmlExporter
  └─ per track: CuePointEntity list
        ├─ Type = Drop/Intro/Outro → POSITION_MARK Type="0" (Hot Cue), Num=0..7
        ├─ Type = Loop              → POSITION_MARK Type="4" (Hot Loop), Num=0..7
        └─ Others                   → Memory Cue (no Num/hot pad)
```

## Waveform Control Rendering Pipeline

`CueForgeWaveformControl` renders 10 layers in order (back → front):

1. **Background** — solid `#0A0E27`
2. **Phrase map strip** — bottom 12px colored bands per `PhraseSegment` label (Intro=blue, Drop=red, Breakdown=purple, etc.)
3. **Beat grid** — bar lines (bright) + beat subdivision lines (dim), spaced by BPM
4. **Onset density fill** — translucent yellow/orange fill from `OnsetDensityCurveData`, shows rhythmic activity density
5. **RGB waveform** — `WaveformLow=Red`, `WaveformMid=Green`, `WaveformHigh=Blue` (Rekordbox convention), drawn as vertical lines per sample column
6. **Energy overlay** — bright green line from `EnergyCurveData` (RMS envelope)
7. **Vocal overlay** — red tint strips where `VocalDensityCurveData > threshold`
8. **Loop blocks** — translucent shaded region with draggable `|` edge handles at `LoopInSeconds`/`LoopOutSeconds`
9. **Cue markers** — vertical colored lines + triangle flags per `OrbitCue`, color-coded by `CueRole`
10. **Playhead** — bright cyan vertical line at `CurrentPlayPosition`

**Interaction:**
- Drag a cue marker (within 18px hit radius) → updates `OrbitCue.Timestamp` live, snapped by `ApplySnapping`
- Drag loop edge handles → updates `LoopInSeconds`/`LoopOutSeconds`
- `SnapToGrid` toggle + `QuantizeBeatString` combo drive `ApplySnapping` resolution

## State Management

### Working Draft Isolation

All edits operate on `WorkingCues` (in-memory `ObservableCollection<OrbitCue>`). Nothing writes to SQLite until **✓ Commit** is clicked.

- **Discard** → re-fetches `CuePointEntity` from DB and restores `WorkingCues`
- **HasUncommittedChanges** — set by `WorkingCues.CollectionChanged`; controls visibility of Commit/Discard buttons and the "● UNSAVED CHANGES" status badge

### Undo / Redo

- Max 50 snapshots (`MaxUndoHistory`)
- Each mutating operation calls `PushSnapshot()` before changing `WorkingCues`
- Snapshot = `OrbitCue[]` deep copy (all fields)
- Redo stack is cleared on any new edit

### Session Persistence

`AppConfig.CueForgeLastTrackHash/Title/Artist` — written to config JSON on every `LoadTrackAsync`. On next launch, `CueForgeViewModel` constructor restores the last track immediately.

## Key Identifiers

| Concept | Identifier |
|---------|------------|
| Track identity | `TrackUniqueHash` (SHA256 of file content) |
| Cue slot mapping | `CuePointEntity.SlotIndex` 0–7 = Hot Cue pads, -1 = Memory Cue |
| Cue types | `CuePointType`: `Intro=0, Outro=1, Drop=2, Breakdown=3, Build=4, PhraseBoundary=5` |
| Cue roles (in-memory) | `CueRole`: `Intro, KickIn, Breakdown, Breakdown2, Drop, Climax, Vocals, Outro, Bridge, Build, PhraseStart, Custom` |
| Loop flag | `IsLoop=true` + `LoopEndSeconds` on the same entity |
| User vs auto cue | `CueSource.User` — never overwritten by Auto-Generate; `CueSource.Auto` — replaced on re-generate |

## Extension Points

To further develop Cue Forge, the highest-leverage areas are:

- **`CueForgeWaveformControl`** — all visual features (zoom, scroll, waveform zoom-in, waveform color schemes) live here. Rendering is pure GDI-via-Avalonia `DrawingContext`, no external libs.
- **`Engine/Cueing/CueGenerationService.cs`** — add a Path 4 or improve Path 2/3. All paths receive the same `AnalysisPipelineResult` so new signals just need to be added there.
- **`BuildAnalysisResultFromFeatures`** in `CueForgeViewModel` — the bridge between stored `AudioFeaturesEntity` columns and `AnalysisPipelineResult`. When new analysis columns are added to `AudioFeaturesEntity`, wire them here for the Auto-Generate fast path.
- **`CuePointEntity` + migration** — to store new cue metadata (e.g., confidence, phrase label, color scheme) add columns here and create a new EF migration.
- **`RekordboxXmlExporter`** — to change how cues export to DJ software (e.g. Pioneer CDJ, Engine DJ, Serato), start here.
